### Learning 1 — Install-path vs repo-path divergence

- **Type:** gotcha
- **Title:** "Edit the repo path; the install path is read-only after install"
- **Problem:** When the agent runs `pwrl init` or `npm install`, a copy of `pwrl-*/` from the repo is placed in `~/.agents/skills/`. Editing the install path does **not** propagate to the repo, and the next `npm install` will overwrite the edits. The agent made all 5 SKILL.md edits for plan 2026-06-29-003 (U1–U5) in `~/.agents/skills/pwrl-*/` and committed the wrong files. The shipped package (the repo) contained none of the changes.
- **Application:** Before editing any file, verify which tree is the source of truth. For PWRL, the source of truth is the repo at `~/Projects/pwrl/pwrl-*/`. The install at `~/.agents/skills/pwrl-*/` is a deployment artifact.
- **Source Context:** review of `7dcb487 feat: enforce task lifecycle contract across pwrl-work and pwrl-review` — diff showed only docs+CHANGELOG+package.json; 5 SKILL.md files unchanged in repo but modified in `~/.agents/skills/`.
- **Confidence:** high
- **Linked Issues:** N/A (local-only incident)

### Learning 2 — Code-edit Pre-Flight Guard gap

- **Type:** pattern
- **Title:** "Extend the Pre-Flight Guard contract to code-edit location, not just task file transitions"
- **Problem:** The contract established by U1–U5 (in `pwrl-work/SKILL.md` §"Task Lifecycle Contract") defines who owns each task file transition. It does **not** define who owns the editing of source code files. The agent treated code edits as "unowned" and edited the install path (allowed) instead of the repo path (required). The contract itself was the source of the bug: it implicitly assumed the agent already knew the right location.
- **Application:** When a contract describes who may perform an action, it should also describe **where** that action takes place. For PWRL, the "where" is "the repo tree, never the install tree." Add a Code-Edit Pre-Flight Guard to `pwrl-work/SKILL.md` (or a new section) that asserts "all edits to `pwrl-*/SKILL.md` files must target the repo path, not the install path." Optionally add the inverse to the postinstall script (refuse to copy if the install path is newer than the repo path on the same file).
- **Source Context:** Cross-reference to `task-state-machine-enforcement-2026-06-29.md` and `cross-skill-contract-enforcement-2026-06-29.md` — both cover task file ownership but not code file ownership.
- **Confidence:** high
- **Linked Issues:** N/A

### Learning 3 — Verify acceptance criteria against the repo, not the install

- **Type:** gotcha
- **Title:** "Always grep the repo path; the install can lie"
- **Problem:** The acceptance-criteria verification commands in plan 2026-06-29-003 (and in the tasks U1–U7) check files at `/home/wicttor/.agents/skills/pwrl-*/SKILL.md`. If run against the install path, the agent will report "PASS" because the changes ARE there — but the published npm package (which uses the repo path) does not contain them. Verification must target the **shipping** tree.
- **Application:** All `pwrl-*/` file paths in `Acceptance Criteria` and `Verification Commands` sections of plans and tasks must use the repo path. Use a consistent pattern: `~/Projects/pwrl/pwrl-*/` (or `pwd` in the repo root, or a `${REPO_ROOT}` variable in tasks). Audit existing tasks to find any that still reference the install path.
- **Source Context:** The 7 task files for plan 2026-06-29-003 all declare `files: [/home/wicttor/.agents/skills/.../SKILL.md]` in their frontmatter. The agent followed the file list literally and edited the install path.

- **Title:** "Verify commit message against the diff before merging"

Mismatches block the push.

### Learning 6 — Line-count standard drift in task acceptance criteria

- **Type:** decision
- **Title:** "Task acceptance criteria must reference the current OKF line-count standard, not a historical one"
- **Problem:** Tasks U2/U3/U4 use the line-count thresholds `≤ 350` and `≤ 500` lines, which were the old standards. The OKF standard was relaxed on 2026-06-21 (`pwrl-standards/SCHEMA.md` line 44) to **300 lines** (acceptable range 80–300). The tasks are written against the old standard. The repo files are 431, 516, and 297 lines respectively — over 300 in two cases — but the task says "within standard" by the old 350/500 threshold.
- **Application:** When authoring or reviewing tasks, the line-count acceptance criterion should reference the standard in `pwrl-standards/SCHEMA.md` (or the equivalent current standard doc) by name or path, not a hard-coded number. This makes the criterion self-updating when the standard changes. Alternatively, the standard should expose a "current line-count limit" as a queryable value (e.g., a script) that tasks can reference.
- **Source Context:** `pwrl-standards/SCHEMA.md` line 44: "Acceptable range: 80-300 lines (temporarily relaxed from 170; see note below)". Tasks U2/U3/U4 use 350/500.
- **Confidence:** medium

### Learning 7 — Extend the contract gap decision to file-location scope

- **Type:** pattern (update to existing decision `documented-vs-implemented-contract-gap-2026-06-29.md`)
- **Title:** "Documented-vs-implemented gap applies to file location, not just skill implementation"
- **Problem:** The existing decision learning `documented-vs-implemented-contract-gap-2026-06-29.md` covers the gap between "documented in a learning" and "implemented in a SKILL.md". This review revealed the same pattern one level deeper: a change can be documented in a task AND implemented in a SKILL.md AND be visible in the install — but still be missing from the published package, because the implementation landed in the wrong file tree. The "implementation layer" is not binary (learning ↔ SKILL.md); it's a chain (learning → task → repo path → install path → npm package).
- **Application:** Update `documented-vs-implemented-contract-gap-2026-06-29.md` (or add a sibling) to introduce the **implementation-layer chain**: learning → plan → task → repo → install → published. A change is "fully implemented" only if it propagates through every layer. Add a layer-by-layer audit checklist that the reviewer can run to verify propagation.
- **Source Context:** The pwrl-review on 2026-06-30 rejected plan 2026-06-29-003 because the implementation stopped at the install layer and never reached the repo or published package.
