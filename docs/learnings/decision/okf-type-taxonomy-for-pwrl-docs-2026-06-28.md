# OKF Type Taxonomy Design for PWRL Agent-Generated Documents

## Context

When migrating PWRL-generated documents to OKF v0.1 compliance, we needed to choose `type` values for the frontmatter of every concept document. OKF requires a non-empty `type` field but deliberately does not prescribe values — producers pick descriptive, self-explanatory names. The question was: what type values should PWRL use, and how should they relate to the existing PWRL document category system?

## Decision

**Use `PWRL X` prefixed type values** rather than generic type names.

### Chosen Type Values

| PWRL Category | Directory | OKF `type` Value |
|---------------|-----------|-----------------|
| Learnings | `docs/learnings/` | `PWRL Learning` |
| Tasks | `docs/tasks/` | `PWRL Task` |
| Analysis | `docs/analysis/` | `PWRL Analysis` |
| Guides | `docs/guides/` | `PWRL Guide` |
| Examples | `docs/examples/` | `PWRL Example` |
| Plans | `docs/plans/` | `PWRL Plan` |
| Test Plans | `docs/test-plans/` | `PWRL Test Plan` |

### Rationale

1. **Namespace isolation:** The `PWRL` prefix prevents collisions if non-PWRL documents are added to the same bundle later. A generic `Learning` type could conflict with another tool's learning documents.
2. **Self-documenting:** The prefix immediately identifies the provenance of the document — it was generated by the PWRL skill set.
3. **Descriptive:** Follows OKF's guidance that types should be "descriptive and self-explanatory." `PWRL Learning` is clearer than just `Learning` or an opaque code like `LRN-001`.
4. **Consistent with OKF philosophy:** OKF encourages producers to pick meaningful values without central registration. `PWRL X` is meaningful within the PWRL ecosystem.

### Field Mapping Decisions

When adding OKF frontmatter, we mapped existing PWRL field names:

| PWRL Field | Action | OKF Field |
|------------|--------|-----------|
| `date` | **Rename** | `timestamp` (OKF recommended, ISO 8601) |
| `category` | **Preserve as custom** | N/A (producer-defined extension, OKF permits) |
| `title` | **Keep** | `title` (OKF recommended) |
| `tags` | **Keep** | `tags` (OKF recommended) |
| `severity` | **Preserve as custom** | N/A |
| `domain` | **Preserve as custom** | N/A |
| `status` | **Preserve as custom** | N/A |
| `source` | **Preserve as custom** | N/A |
| (none) | **Add** | `type` (OKF required) |
| (none) | **Add where missing** | `description` (OKF recommended) |

### Why Not Generic Types?

**Alternative considered:** Use generic type names like `Learning`, `Task`, `Analysis` without the `PWRL` prefix.

**Rejected because:**
- No namespace protection: another tool in the same bundle could use the same type name
- Less self-documenting: a consumer can't tell if a `Learning` document came from PWRL or another system
- Ambiguous in cross-organization exchange: "Learning" means different things to different teams

### Why Not Hierarchical Types?

**Alternative considered:** Use hierarchical type names like `PWRL/Learning` or `pwrl.learning`.

**Rejected because:**
- OKF types are flat strings, not hierarchical paths
- The directory structure already provides hierarchy (`docs/learnings/decision/`)
- Slashes or dots in type names could confuse YAML parsers or downstream consumers

## Quality Implications

### Benefits
- **Interoperability:** Any OKF consumer can route/filter by type without understanding PWRL internals
- **Extensibility:** New PWRL document categories can get new `PWRL X` types without changing existing ones
- **Grepability:** `grep "type: PWRL"` finds all PWRL-generated documents instantly

### Trade-offs
- More verbose than generic types (`PWRL Learning` vs `Learning`)
- Requires consensus on prefix for consistency across the project
- Could be seen as "branding" in what should be an open format

## Implementation

The taxonomy is documented in `docs/OKF-TYPES.md` (created by U1 of the migration plan). This serves as the single source of truth for all PWRL type values.
