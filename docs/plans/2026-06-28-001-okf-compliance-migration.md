# OKF Compliance Migration for PWRL Documents (Standard)

**Date:** 2026-06-28 | **Type:** refactor

## Overview

The PWRL project contains ~85 concept documents across 7 categories (learnings, tasks, analysis, guides, examples, plans, test-plans) plus 2 index files, all generated by PWRL skills. These currently lack OKF (Open Knowledge Format) v0.1 compliance: only ~12 have the required `type` frontmatter field, 19 files lack frontmatter entirely, existing field names don't align with OKF conventions, and `index.md` files don't follow OKF §6 structure. This plan migrates all PWRL-generated documents to OKF v0.1 conformance — metadata enrichment only, no semantic content changes.

## Key Technical Decisions

- **Type Taxonomy:** Define dedicated PWRL-specific `type` values rather than generic ones, keeping them descriptive as OKF recommends (e.g., `PWRL Learning` not just `Learning`). This avoids collisions with non-PWRL documents in the same bundle.
- **Bulk Multi-Replace Pattern:** Use the proven multi-replace batch approach (from `bulk-metadata-sync` learning) for groups of 5+ files sharing the same edit pattern. Individual edits for outliers.
- **Field Mapping:** Map `date` → `timestamp` (OKF recommended name), preserve `category` as a custom field (OKF allows producer-defined extensions), keep `title`, `tags`, `severity`, `domain`, `status` intact.
- **Index Non-Destructive Update:** Back up existing `INDEX.md` files before overwriting with OKF §6 structure, following the `non-destructive-index-regeneration` learning.
- **No Content Changes:** Absolutely no changes to body content (markdown after frontmatter closing `---`). This is a pure metadata enrichment migration.
- **Backup Files Excluded:** Files in `docs/learnings/.backups/` are explicitly excluded from OKF compliance (archive material, not active concepts).

  - **Approach:** Map each PWRL document category to an OKF `type` value. Root index enumerates subdirectories with descriptions following OKF §6 progressive disclosure pattern. Root index may include frontmatter with `okf_version: "0.1"` (the only place OKF permits frontmatter in an `index.md`).
  - **Test Scenarios:**

## System-Wide Impact

- **No API or Code Changes:** This is purely a documentation metadata migration. No code, build, or deployment changes.
- **Version Control:** All changes are in the `docs/` directory. Git history provides full rollback capability. Recommended: single commit per unit or one consolidated commit for the migration.
- **Downstream Consumers:** Any tools or agents that parse PWRL document frontmatter (e.g., learnings index generators, task trackers) will need to handle the new `type` field and `date`→`timestamp` rename. These consumers already tolerate unknown fields (PWRL convention).
- **OKF.md Itself:** `docs/OKF.md` already has a `type` field and is OKF-compliant. No changes needed.
- **Performance:** No impact — static file metadata changes only.
