# GitHub PR Sync Protocol

Technical specification for synchronizing pwrl-review findings to GitHub pull requests.

## Overview

PWRl-review-sync-status bridges the review workflow and GitHub by:
1. Posting detailed findings as PR comment
2. Creating formal GitHub review (APPROVE/REQUEST_CHANGES)
3. Updating PR labels based on verdict and quality scores
4. Maintaining audit trail of all GitHub interactions

## GitHub API Reference

### Authentication

**Methods:**
- **GitHub CLI (`gh`):** Automatic auth using `~/.config/gh/hosts.yml`
- **PAT (Personal Access Token):** Manual header auth
- **OAuth:** Application-level access (requires app credentials)

**Required Scopes:**
- `repo` — Full repo access (read/write PRs, reviews, comments)
- `workflow` — GitHub Actions (optional, for status checks)

**Token Expiry:**
- PAT tokens: No expiry (or user-specified expiry)
- OAuth tokens: Typically 8 hours
- Always check token validity before API calls

### Rate Limits

- **Unauthenticated:** 60 requests/hour
- **Authenticated (PAT):** 5000 requests/hour
- **GitHub App:** 15000 requests/hour

**Headers returned:**
```
X-RateLimit-Limit: 5000
X-RateLimit-Remaining: 4999
X-RateLimit-Reset: [unix-timestamp]
```

**Retry Strategy:**
- Check `X-RateLimit-Remaining` before each call
- If ≤ 100 remaining: Pause and warn user
- On 403 (rate limit hit): Use exponential backoff
  - Wait 1s, retry
  - Wait 2s, retry
  - Wait 4s, retry
  - After 3 retries: Fail and save locally

### API Endpoints Used

**Post PR Comment:**
```
POST /repos/{owner}/{repo}/issues/{issue_number}/comments
Content-Type: application/json

{
  "body": "[markdown comment]"
}

Response: {
  "id": 12345,
  "url": "https://api.github.com/repos/owner/repo/issues/comments/12345",
  "body": "[comment]"
}
```

**Create PR Review:**
```
POST /repos/{owner}/{repo}/pulls/{pull_number}/reviews
Content-Type: application/json

{
  "body": "[review summary]",
  "event": "APPROVE" | "REQUEST_CHANGES" | "COMMENT"
}

Response: {
  "id": 67890,
  "pull_request_url": "...",
  "state": "APPROVED" | "CHANGES_REQUESTED" | "COMMENTED"
}
```

**Update PR Labels:**
```
PATCH /repos/{owner}/{repo}/issues/{issue_number}/labels
Content-Type: application/json

{
  "labels": ["review-approved", "coverage-low", ...]
}

Response: [
  { "name": "review-approved", "color": "green" },
  ...
]
```

**Get PR Details:**
```
GET /repos/{owner}/{repo}/pulls/{pull_number}

Response: {
  "number": 123,
  "title": "...",
  "state": "open",
  "labels": [...],
  ...
}
```

## Label Management

### Standard Labels

**Review Status (mutually exclusive):**
- `review-approved` — Code approved, ready to merge
- `review-changes-requested` — Fixes required before merge
- `review-rejected` — Major blockers, discussion needed

**Quality Issues (can combine):**
- `security-concerns` — Security issues found
- `coverage-low` — Test coverage below threshold
- `docs-incomplete` — Documentation gaps
- `build-failing` — Build or test failures
- `code-quality` — Code quality issues

### Label Creation

If label doesn't exist, create with standard format:
- Name: `kebab-case`
- Color:
  - Review status: green (#28a745), yellow (#fbca04), red (#d73a49)
  - Quality issues: orange (#ed7923), purple (#a371f7)
- Description: "Auto-generated by pwrl-review"

### Application Logic

```python
def compute_labels(verdict, findings):
    labels = []

    # Verdict labels (remove old, add new)
    if verdict == "APPROVED":
        labels.append("review-approved")
        remove_labels(["review-changes-requested", "review-rejected"])
    elif verdict == "REQUEST_CHANGES":
        labels.append("review-changes-requested")
        remove_labels(["review-approved", "review-rejected"])
    else:  # REJECTED
        labels.append("review-rejected")
        remove_labels(["review-approved", "review-changes-requested"])

    # Quality labels (add if issues found)
    if any(f.category == "security" and f.severity == "CRITICAL" for f in findings):
        labels.append("security-concerns")

    if any(f.category == "tests" and "coverage" in f.description for f in findings):
        labels.append("coverage-low")

    if any(f.category == "documentation" for f in findings):
        labels.append("docs-incomplete")

    if any(f.category == "integration" for f in findings):
        labels.append("build-failing")

    return labels
```

## PR Number Resolution

### Input Sources

**Method 1: Direct argument**
- User provides PR number: `/pwrl-review-sync-status 123`
- Extract number: `pr_number = 123`

**Method 2: Context from branch**
- Branch follows pattern: `feature/github-<pr-number>-*`
- Extract: `pr_number = extract_pr_from_branch_name()`

**Method 3: GitHub API search**
- Find open PR for current branch:
  ```
  GET /repos/{owner}/{repo}/pulls?head={owner}:{branch}&state=open
  ```
- Use first result if exactly 1 match

**Method 4: Manual fallback**
- If all automatic methods fail: Ask user for PR number
- Prompt: "Enter GitHub PR number for this review:"

### Owner/Repo Detection

**Priority order:**
1. Parse from remote `origin` URL:
   ```
   git remote get-url origin
   → https://github.com/owner/repo.git
   → owner/repo
   ```

2. Check `github.repository` config:
   ```
   git config github.repository
   ```

3. Ask user:
   ```
   "Enter repo as owner/repo (e.g., microsoft/vscode):"
   ```

## Error Handling & Recovery

### Transient Errors (Retry)

| Error | Status | Action |
|-------|--------|--------|
| Network timeout | 0 | Retry with backoff |
| Rate limit | 429 | Wait until reset, retry |
| Service unavailable | 503 | Wait 60s, retry up to 3x |
| Temporary failure | 5xx | Exponential backoff, retry 3x |

**Retry Logic:**
```
for attempt in 1..3:
  try:
    result = api_call()
    return result
  except TransientError as e:
    wait = 2 ^ (attempt - 1) seconds
    log("Retrying in {wait}s...")
    sleep(wait)
  except FinalError as e:
    raise e
```

### Permanent Errors (Fallback)

| Error | Status | Fallback |
|-------|--------|----------|
| Invalid token | 401 | Request re-auth |
| Insufficient perms | 403 | Request scope upgrade |
| PR not found | 404 | Prompt user for correct PR number |
| Invalid repo | 404 | Prompt user for repo name |
| Comment too large | 422 | Truncate + link to artifact |

**Fallback Strategy:**
1. Try GitHub API posting
2. On permanent error: Save artifact locally
3. Provide user with instructions for manual posting
4. Log error for debugging

### Local Fallback Storage

If GitHub sync fails:
1. Save report artifact to `.context/pwrl-review/sync-failed/` directory
2. Generate human-readable summary
3. Provide instructions to user:
   ```
   GitHub API failed. Review findings saved locally.

   File: /path/to/artifact.md

   To post manually:
   1. Copy findings from above file
   2. Go to: https://github.com/owner/repo/pull/123
   3. Paste as comment
   4. Leave a "REQUEST_CHANGES" review
   ```

## Audit Trail

### Logging

All GitHub interactions logged with:
- Timestamp (ISO-8601)
- API endpoint called
- Request body (sanitized)
- Response status and headers
- Any errors or retries
- Final outcome (success/fallback)

**Log Example:**
```
2026-06-24T14:30:00Z | POST /repos/owner/repo/issues/123/comments | 201 | Comment ID: 98765
2026-06-24T14:30:02Z | POST /repos/owner/repo/pulls/123/reviews | 200 | Review ID: 54321
2026-06-24T14:30:03Z | PATCH /repos/owner/repo/issues/123/labels | 200 | 3 labels updated
```

### Metrics

Track for each run:
- Comments posted (count)
- Reviews created (count)
- Labels updated (added/removed)
- Errors encountered (with recovery method)
- Time to post (seconds)
- GitHub API calls used (of quota)

## Edge Cases

### Existing Reviews

If PR already has a review from same reviewer:
- Option 1: Update existing review (dismiss old, post new)
- Option 2: Post comment only (avoid duplicate reviews)
- **Recommended:** Option 2 (post comment only)

### Comment Size Limit

GitHub comments limited to 65,536 characters (~60KB markdown).

**If findings too large:**
1. Truncate findings to top 20
2. Add footer: "Findings truncated. See full report: [link]"
3. Post comment successfully
4. Log truncation event

### Deleted PR

If PR deleted between Phase 4 and Phase 5:
- GitHub returns 404 (PR not found)
- Fall back to local artifact save
- Notify user: "PR 123 not found. Findings saved locally."

### Multiple Reviews Per PR

If re-running review on same PR:
1. Check if previous review exists from same reviewer
2. If yes: Post comment (update information)
3. If no: Create new formal review + comment
4. Never create duplicate formal reviews
