# Add Email Validation to Signup

**Goal:** Validate email format on signup form submission to prevent invalid emails from entering the system.

## Context

The current signup flow accepts any string as an email, causing downstream issues:
- Failed email deliveries (20% bounce rate)
- Database pollution with malformed emails
- Support burden from users entering invalid emails

This task adds client-side and server-side validation using the existing validator pattern established in `src/utils/validators.ts`.

**Non-goals:** Email verification (sending confirmation emails) - that's U3

### Pattern 3: Retry Logic with Exponential Backoff

```typescript
// Common retry pattern in codebase
async function withRetry<T>(
  fn: () => Promise<T>,
  maxRetries: number = 3
): Promise<T> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      if (attempt === maxRetries - 1) throw error;
      await sleep(Math.pow(2, attempt) * 1000); // Exponential backoff
    }
  }
  throw new Error('Max retries exceeded');
}
```

## API References

### SendGrid API

- **Endpoint:** `POST https://api.sendgrid.com/v3/mail/send`
- **Auth:** `Authorization: Bearer {API_KEY}`
- **Request:**
  ```json
  {
    "personalizations": [
      {
        "to": [{"email": "recipient@example.com"}]
      }
    ],
    "from": {"email": "sender@example.com"},
    "subject": "Email subject",
    "content": [
      {
        "type": "text/plain",
        "value": "Email body"
      }
    ]
  }
  ```
- **Success Response:** `202 Accepted`
  ```json
  {
    "message_id": "abc123..."
  }
  ```
- **Error Codes:**
  - `400`: Bad request (invalid email format)
  - `401`: Unauthorized (invalid API key)
  - `429`: Rate limit exceeded
  - `500`: SendGrid server error
- **Documentation:** https://docs.sendgrid.com/api-reference/mail-send/mail-send

### @sendgrid/mail Library

- **Installation:** `npm install @sendgrid/mail`
- **Setup:**
  ```typescript
  import sgMail from '@sendgrid/mail';
  sgMail.setApiKey(process.env.SENDGRID_API_KEY);
  ```
- **Send Email:**
  ```typescript
  const msg = {
    to: 'recipient@example.com',
    from: 'sender@example.com',
    subject: 'Subject',
    text: 'Plain text body',
    html: '<p>HTML body</p>'
  };
  await sgMail.send(msg);
  ```
- **Error Handling:**
  ```typescript
  try {
    await sgMail.send(msg);
  } catch (error) {
    if (error.response) {
      console.error(error.response.body);
    }
  }
  ```

## Template Selection in Practice

When generating tasks, pwrl-tasks should:

1. Check number of files affected (1 = simple, 2-4 = standard, 5+ = complex)
2. Check for external integrations (APIs, libraries = complex)
3. Check implementation approach description length (short = simple, detailed = standard/complex)
4. Check test scenarios count (<3 = simple, 3-5 = standard, 6+ = complex)
5. Default to standard if uncertain

The goal is to provide enough detail for pwrl-work to execute without constant clarification, but not so much detail that it becomes prescriptive or overwhelming.
