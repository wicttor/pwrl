## Template Selection Guide

| Task Complexity                           | Template                       | Sections to Include                                 |
| ----------------------------------------- | ------------------------------ | --------------------------------------------------- |
| **Simple** (1-2 files, clear approach)    | Minimal                        | Goal, Implementation, Testing, Acceptance Criteria  |
| **Standard** (3-5 files, some complexity) | Full (skip API section if N/A) | All sections except API if not needed               |
| **Complex** (6+ files, integration, APIs) | Full                           | All sections, detailed Code Patterns and Edge Cases |

---

## Frontmatter Fields Reference

| Field            | Required | Format                                       | Purpose                           |
| ---------------- | -------- | -------------------------------------------- | --------------------------------- |
| `unit-id`        | Yes      | `U1`, `U2`, etc.                             | Stable identifier from plan       |
| `plan`           | Yes      | Relative path                                | Link back to source plan          |
| `status`         | Yes      | `to-do`, `in-progress`, `for-review`, `done` | Current task state                |
| `created`        | Yes      | `YYYY-MM-DD`                                 | When task was generated           |
| `dependencies`   | Yes      | YAML array of unit IDs                       | Tasks that must complete first    |
| `files`          | Yes      | YAML array of paths                          | Files this task will modify       |
| `assignee`       | No       | String                                       | Person working on this (optional) |
| `estimated-time` | No       | String                                       | Time estimate (optional)          |
| `priority`       | No       | `high`, `medium`, `low`                      | Task priority (optional)          |

## Best Practices

1. **Right-size the template**: Simple tasks don't need every section
2. **Concrete examples**: Show actual code, not pseudocode
3. **Link liberally**: Connect to files, docs, and other tasks
4. **Test-first structure**: Make it easy to write tests before implementation
5. **Self-contained**: Task should be executable without constant plan references
6. **Clear acceptance criteria**: No ambiguity about "done"

---

## Anti-patterns

❌ **Too vague**: "Update the authentication logic" without specifics
❌ **Too prescriptive**: Writing production code in the task (that's the work, not the plan)
❌ **Missing context**: No explanation of why or how this fits in
❌ **No verification**: Unclear how to test or confirm completion
❌ **Dependency hell**: Too many interdependencies between tasks
